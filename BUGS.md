1. Некорректный формат ответа при создании объявления

При успешном создании объявления (POST /api/1/item) сервер возвращает JSON с единственным полем status, в котором ID "зашит" внутри текстовой строки. Это противоречит документации, которая обещает полный JSON-объект созданной сущности.

Ожидаемый JSON:
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "sellerId": 123456,
  "name": "Товар",
  "price": 10000,
  "statistics": { ... },
  "createdAt": "..."
}

Фактический JSON:
{
  "status": "Сохранили объявление - 3fa85f64-5717-4562-b3fc-2c963f66afa6"
}


2. Метод GET /item/:id возвращает массив вместо объекта

Запрос конкретного объявления по его уникальному ID должен возвращать один JSON-объект. Вместо этого API возвращает массив, содержащий один объект. Это нарушает REST-конвенции и усложняет обработку ответа на стороне клиента.

Ожидаемый результат: Объект { ... }
Фактический результат: Массив [{ ... }]


3. Метод GET /statistic/:id возвращает массив вместо объекта

Аналогично дефекту BG-02, запрос статистики для одного объявления возвращается в виде массива объектов, а не одного объекта.

Ожидаемый результат: Объект { "likes": 0, ... }
Фактический результат: Массив [{ "likes": 0, ... }]
